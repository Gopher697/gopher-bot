# Workbench Index

## Start Here

New agents should use this reading order:

1. Read this file first.
2. Read `PROJECT_REGISTRY.md` to understand registered projects, entrypoints, and authority boundaries.
3. Read `KNOWLEDGE_SPACES.md` when the task needs broader wiki/knowledge-layer orientation.
4. Read the relevant project entrypoint named in `PROJECT_REGISTRY.md`.
5. Read Workbench-wide SOPs only from `D:\gopher-workbench\sops\` unless another SOP has been explicitly promoted or mirrored there.
6. Treat session notes, archives, staging folders, raw imports, pasted reports, and old handoffs as historical/reference unless a current-state file promotes them.

## Authority Order

- Human instructions in the current session override stale notes.
- `PROJECT_REGISTRY.md` defines project boundaries and entrypoints for agent orientation.
- Project entrypoint/current-state files define current authority only for their own project.
- `D:\gopher-workbench\sops\` defines Workbench-wide SOP authority.
- GopherVault SOPs and project-local SOPs may be useful references or promotion candidates, but are not automatically Workbench-wide authority.
- More comprehensive does not mean more authoritative.
- Session notes are historical by default.
- Staging/raw/archive/pasted materials are evidence/reference by default.
- Ambiguous authority should be flagged, not silently resolved.

## Core Workbench Files

- `WORKBENCH_INDEX.md`: First-read orientation file for Workbench authority and navigation.
- `PROJECT_REGISTRY.md`: Human/agent-readable map of registered projects, entrypoints, and authority boundaries.
- `KNOWLEDGE_SPACES.md`: Map of broader wiki/knowledge layers, including the Workbench control layer, MCP configured access, project workspaces/repos, project-specific wikis, GopherVault external memory, SOP authority and candidates, and session/staging/archive/reference material. It does not replace `PROJECT_REGISTRY.md`.
- `README.md`: Workbench setup, safety model, MCP tools/resources, and local operational notes.
- `PROJECT.md`: Configured project summary for the Workbench repo itself.
- `sops\`: Workbench-wide SOP authority unless another SOP has been explicitly promoted or mirrored there.
- `notes\sessions\`: Historical session logs by default, not canonical current state.

## How To Enter A Project

1. Look up the project in `PROJECT_REGISTRY.md`.
2. Read the listed actual first file.
3. Check open authority questions before relying on old notes.
4. Use project-local SOPs only within that project unless promoted.
5. Do not import assumptions from one project into another.

## What Not To Trust Automatically

- Session notes that contain current-seeming facts.
- Staging folders.
- Raw imports.
- Archived reports.
- Pasted handoffs.
- Old Codex/Claude outputs.
- GopherVault notes that are more detailed than Workbench files.
- Project-specific SOPs that look reusable but have not been promoted.

## When To Update This Index

Update this file only when:

- Workbench orientation changes.
- The project registry path or role changes.
- SOP authority rules change.
- The project-entry workflow changes.
- New major connective-tissue files are added.

Do not update this file for ordinary project progress. Ordinary project progress belongs in the relevant project current-state file or project notes.
